# 编剧 Agent 工位产物

## 目标理解

作为串行多 Agent 视频工作流的编剧角色，本工位的核心任务是基于上游产品 Brief 生成故事主线和核心冲突，为后续分镜、拍摄脚本提供叙事框架。需要确保故事既能准确传达 HarnessOS V4.x 的 Headless Workflow Core 产品定位，又能在有限时间内呈现必须出现的四个桥段，同时保持现代 SaaS 产品介绍片的节奏感和可审计性主题。

## 关键输出

### 1. 故事主线

视频采用三幕式结构，以「可控的智能化」为核心主题贯穿全片：

**第一幕：困境与抉择（0:00-0:30）**
以一个典型研发团队的 AI 工作流搭建场景开篇，团队面临复杂流程难以管理、审计困难的痛点，引出「我们需要一个既智能又可控的工作流平台」的核心诉求。

**第二幕：方案与验证（0:30-1:30）**
展示 HarnessOS Headless 方案的三大核心能力：自然语言定义工作流、多 Head 输出（Drawio、HTML Report、TUI）、工位化制品管理。重点呈现串行 Agent 工位如何协同工作，以及中间工位重跑时的下游状态管理机制。

**第三幕：价值与保障（1:30-2:00）**
回归用户核心关切，展示完整的治理证据链和审计能力，传递「每个变更都可追溯、每次执行都有记录」的品牌承诺。

### 2. 核心冲突

| 冲突维度 | 具体表现 |
|---------|----------|
| 效率 vs 可控 | Agent 自动执行的高效性与人类最终确认的必要性之间的张力 |
| 简洁 vs 透明 | 自然语言输入的易用性与 WorkflowSpec、Drawio 等产出的专业性之间的平衡 |
| 动态 vs 审计 | 工作流运行时产生的数据与静态证据链之间的对应关系 |

### 3. 桥段设计

| 桥段 | 时长 | 内容描述 |
|------|------|----------|
| Headless-first 路线 | 0:35-0:50 | 通过 TUI 界面演示自然语言定义工作流，瞬间生成结构化输出的对比画面 |
| 串行 Agent 工位 | 0:55-1:10 | 以流水线视觉化展示编剧→分镜→生成的串行工位流转过程 |
| 中间重跑与下游 stale | 1:15-1:25 | 演示某一工位重跑后下游标记为 stale 状态，以及触发重新执行的全流程 |
| HTML 报告与证据链 | 1:40-1:55 | 展示 HTML 治理报告中工位产物、质量评分、完整证据链的交互查阅 |

## 输入引用

本工位严格依据以下输入资料构建故事框架：

1. **目标受众**：研发负责人、技术管理者、业务用户的三类视角需要在视频中均有体现，技术管理者侧重审计场景，业务用户侧重报告查阅体验
2. **核心信息五点**：自然语言/TUI 定义 → 多 Head 输出 → 工位制品查看 → 用户确认 → Agent 建议不动作为故事逻辑锚点
3. **视频风格**：现代 SaaS 产品介绍，节奏清晰，强调可审计、可恢复、可解释的主题需要在叙事中自然融入，避免生硬说教
4. **四个必须桥段**：必须出现且需在合理叙事位置自然衔接，不可独立于主线之外

## 质量风险

| 风险类型 | 风险描述 | 可能影响 |
|----------|----------|----------|
| 信息过载 | Headless、多工位、stale 状态等概念密集，可能导致观众理解疲劳 | 观众对核心信息的接受度下降 |
| 桥段割裂 | 四个桥段若单独罗列可能破坏叙事连贯性 | 视频整体流畅度和说服力受损 |
| 技术晦涩 | WorkflowSpec、证据链等专业内容若缺乏语境铺垫 | 非技术受众无法产生共鸣 |
| 时长失衡 | 三幕结构若篇幅分配不均可能导致重心偏移 | 核心价值传递不充分 |

## 下一工位建议

### 输出要求

交付分镜脚本（Shot List），包含每个桥段的具体镜头描述、时长分配、画面构图、旁白文案。建议附注每个镜头对应的工位协作关系（如需调用分镜 Agent 或生成 Agent 的节点）。

### 关键关注点

1. **叙事节奏修正**：根据分镜的实际可执行性，可能需要对故事主线进行微调，建议保留编剧层面的核心冲突设计
2. **视觉化可行性**：串行工位流转、stale 状态等抽象概念需要依赖视觉语言表达，建议在分镜阶段增加 Storyboard 草稿
3. **受众视角检查**：建议在分镜中加入针对三类受众的视角切换提示，确保信息覆盖均衡

# 文案 Agent 工位产物

## 目标理解

基于 HarnessOS Headless 发布视频 Brief，本工位负责生成**视频旁白、标题和短视频文案**，用于向目标受众（研发负责人、技术管理者、业务用户）传达产品核心价值，重点突出：

- Headless-first 架构转型
- 串行 Agent 工位协作模式
- 工作流可审计、可恢复、可解释的特性

## 关键输出

### 视频标题

> **"从代码到掌控：HarnessOS Headless 工作流引擎发布"**

备选标题：
- "AI 工作流的透明之道：HarnessOS Headless 来了"
- "把 AI 工作流装进透明盒子"

---

### 视频旁白文案

**【开场 0:00-0:15】**
> 你们有没有遇到过这种情况？AI 工作流跑了三个小时，结果出来一堆不知道哪儿来的错误，想查日志都不知道从哪下手。

**【引入 0:15-0:45】**
> 今天，HarnessOS 4.x 带来了全新的答案——Headless 工作流引擎。
>
> 你可以用自然语言描述需求，或者用 TUI 拖拽式操作，系统会自动生成标准化的 WorkflowSpec、清晰的 Drawio 流程图，还有完整的 HTML 运行报告。
>
> 每一个工位产出了什么、用了多少 token、质量如何，全部一目了然。

**【核心卖点 0:45-1:30】**
> 更重要的是，这不是黑盒。
>
> 来看这个串行多 Agent 工作流：文案 Agent 生成脚本 -> 图像 Agent 生成配图 -> 审核 Agent 检查合规。如果中间的图像生成出了问题，只需要重跑图像 Agent，下游自动标记为 stale 状态，等你修复后继续运行。全程有据可查。
>
> 所有持久变更都需要你确认。Agent 只提建议，绝不自行执行。

**【收尾 1:30-2:00】**
> 现在，无论是研发负责人想搭建工作流，技术管理者想审计过程，还是业务用户想查看质量报告，都有了统一的入口。
>
> HarnessOS Headless，让 AI 工作流真正可控。
