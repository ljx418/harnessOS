# 工位产物：子阶段架构评审

**角色**：架构  
**工位**：子阶段架构评审  
**场景**：长时工程任务工作流  
**目标**：审查子阶段架构影响  

---

## 1. 目标理解

在子阶段架构评审工位中，架构角色需要验证该子阶段的实现方案是否符合总体架构设计，并评估其对上下游模块及整体系统的影响。具体包括：

- **架构一致性**：子阶段设计是否与总架构评审产出的架构决策保持一致
- **接口契约**：子阶段与其他子阶段/模块的接口定义是否清晰、稳定
- **依赖管理**：外部依赖和内部依赖是否合理，是否引入不必要的耦合
- **非功能性需求**：性能、可扩展性、可维护性、安全性是否符合预期
- **约束合规性**：确认实现方案不违反"不允许自动修改代码"、"不允许自动发布"、"持久操作需用户确认"等核心约束
- **可审计性**：是否为该阶段的产出物建立可追溯的 artifact 和 attempt history

---

## 2. 关键输出

本工位完成后，应产出以下 artifact：

| 产物 | 说明 |
|------|------|
| **子阶段架构评审报告** | 对子阶段设计方案的评估结论，包括通过/有条件通过/拒绝 |
| **架构影响分析** | 该子变更对其他模块、系统边界、数据流的影响范围说明 |
| **接口契约定义** | 与上下游交互的 API/协议定义（如有变更） |
| **可追溯性清单** | 关联至总架构评审的决策点，确保每项设计有据可查 |
| **问题跟踪表** | 评审中发现的问题及其状态（未解决/已解决） |

---

## 3. 输入引用

本工位需要引用以下上游 artifact：

- **产品规划**：明确业务目标与用户 story
- **规格梳理**：功能需求与非功能需求规范
- **项目计划蓝图**：里程碑、时间线、资源计划
- **总架构评审产出**：
  - 总体架构文档
  - 技术选型决策
  - 系统边界定义
  - 各子阶段职责划分
- **子阶段计划审计产出**：
  - 子阶段任务拆分
  - 时间估算
  - 依赖关系图
- **前置子阶段产出**（如有）：如开发实施完成后的验收记录

---

## 4. 质量风险

在子阶段架构评审中需重点关注以下风险：

| 风险类别 | 可能问题 | 影响 |
|----------|----------|------|
| **架构漂移** | 子阶段实现偏离总架构决策，未及时同步更新 | 系统一致性破坏，后期返工成本高 |
| **接口断裂** | 定义的契约与实现不一致，导致下游集成失败 | 开发验收或 E2E 验收阶段阻塞 |
| **约束突破** | 实现方案隐含自动修改代码或自动发布逻辑 | 违反核心约束，流程合规性问题 |
| **循环依赖** | 子阶段之间形成过度依赖，破坏模块化隔离 | 影响后续可重跑性，失败定位困难 |
| **可观测性不足** | 未设计日志/审计点，无法满足"可审计"要求 | 人工确认阶段无法回溯决策依据 |
| **非功能缺陷** | 未考虑峰值负载、异常容错等场景 | 上线后系统稳定性问题 |

---

## 5. 下一工位建议

基于评审结果，建议流向以下工位：

### 若评审通过或有条件通过
- **开发实施**：进入编码实现阶段，按评审通过的架构方案落地
- **代码检视**：可在开发过程中并行进行，重点验证实现与架构一致性

### 若评审拒绝
- **返回子阶段计划审计**：重新评估设计前提，修正架构
