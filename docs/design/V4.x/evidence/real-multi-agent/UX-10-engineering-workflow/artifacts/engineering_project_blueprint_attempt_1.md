# 项目计划蓝图 · 工位产物

## 目标理解

为 HarnessOS 的 Headless-first 工作流平台规划一套可审计、可重跑的长时工程任务执行框架。本工位的核心职责是将端到端的期望流程拆解为具有明确边界和可验证交付物的离散阶段，确保每个阶段的产出可沉淀、失败可恢复、全程可追溯。

---

## 关键输出

### 阶段拆解矩阵

| 阶段序号 | 阶段名称 | 执行角色 | 主要交付物 (Artifact) | 可重跑粒度 |
|:--------:|:---------|:---------|:---------------------|:----------|
| P1 | 产品规划 | PM / Stakeholder | PRD 初稿、需求列表 | 需求级 |
| P2 | 规格梳理 | 业务架构师 | 特性规格说明 (FSD)、用例模型 | 用例级 |
| P3 | 项目计划蓝图 | 计划角色 | 本产物——阶段性拆分方案、里程碑定义 | 蓝图级 |
| P4 | 总架构评审 | 系统架构师 | 架构决策记录 (ADR)、技术选型报告 | 决策级 |
| P5 | 子阶段计划审计 | 计划角色 | 迭代计划、资源估算、风险登记册 | 迭代级 |
| P6 | 子阶段架构评审 | 技术负责人 | 子系统接口协议、数据流设计 | 接口级 |
| P7 | 开发实施 | Dev Team | 实现代码、单元测试、集成测试报告 | 任务级 |
| P8 | 开发验收 | QA / Dev | 测试报告、Bug 跟踪记录 | 任务级 |
| P9 | 代码检视 | Code Reviewer | 检视意见闭环、Code Quality Score | PR 级 |
| P10 | E2E 验收 | Test Engineer | E2E 测试报告、性能基准报告 | 场景级 |
| P11 | 人工确认 | Stakeholder | 最终验收签署、Release Note | 发布级 |

> **注**: 阶段 P1–P3 为计划层，P4–P6 为架构层，P7–P10 为实施与验证层，P11 为闭环确认层。

### 交付物流转规则

- 每个阶段完成后须输出一份 `stage-artifact-manifest.json`，声明该阶段产出的所有文件 URI、校验和 (SHA256) 及元数据
- 下一阶段的启动以上一阶段的 manifest 为入口凭证，实现**_gate-based progression_
- 任何阶段的失败均触发 `attempt-history` 记录：包含输入快照、执行上下文、错误日志、修复动作，需显式归档至持久存储

### 持久操作确认机制

| 操作类型 | 触发时机 | 确认形式 |
|:---------|:---------|:---------|
| 代码提交 (Commit) | P7 结束时 | 用户签核 UI + 电子签名 |
| 测试环境变更 | P8 结束时 | 用户签核 UI + 电子签名 |
| 发布准入决策 | P11 结束时 | 用户最终签署 + Release Note |
