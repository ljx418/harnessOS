**工位目标**：对开发实施阶段产出的代码进行系统性审查，评估代码质量、安全性、可维护性与规范合规性，为后续 E2E 验收和人工确认阶段提供质量准入判断。

**在流程中的定位**：

- 前置工位：开发验收（已通过）
- 后续工位：E2E 验收 → 人工确认

## 质量风险

| 风险类型 | 描述 | 缓解措施 |
|----------|------|----------|
| **回归风险** | 新增代码引入旧功能缺陷 | 需提供完整回归测试用例 |
| **安全风险** | 未处理用户输入/权限校验缺失 | 需补充安全审计报告 |
| **兼容性风险** | API 变更未向前兼容 | 需版本管理策略文档 |
| **技术债务** | 遗留代码影响扩展性 | 需标记并记录债务清单 |

## 下一工位建议

**流入条件**：代码检视通过（P0/P1 问题清零）

**建议内容**：

1. **E2E 验收**：基于检视通过的代码，执行端到端集成测试，验证完整业务流程
2. **人工确认**：准备检视报告与问题清单，由人工决策者最终批准进入生产发布
3. **循环治理**：若存在 P2/P3 问题，纳入技术债务登记，延后处理

**注意事项**：

- 根据「所有持久操作都必须用户确认」的约束，E2E 验收通过后仍需人工确认方可进入发布阶段
- 保留 attempt history，确保失败阶段可重跑追溯
