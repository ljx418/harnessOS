# 开发实施 - 工位产物

## 目标理解

作为开发实施工位，本阶段的核心目标是**将经过评审的子阶段架构方案转化为可运行的工程任务 artifact，并确保整个执行流程具备可审计性和可重跑性**。

具体而言，需要完成以下任务：

- 按照子阶段架构评审确定的方案进行代码实现
- 生成符合 HarnessOS 规范的工程任务 artifact（配置文件、执行脚本、元数据等）
- 确保每个执行步骤都有明确的审计日志
- 实现失败重跑机制，保留每次 attempt 的历史记录

本工位承接来自「子阶段架构评审」工位的输出，并将结果交付给「开发验收」工位进行验证。

---

## 关键输出

| 输出项 | 说明 |
|--------|------|
| **工程任务 Artifact** | 包含完整的配置文件、执行脚本、依赖声明等，用于描述一个可执行的工程任务单元 |
| **Attempt History 记录** | 记录每次执行尝试的元数据，包括时间戳、输入参数、执行结果、失败原因（如有） |
| **审计日志** | 按时间顺序记录所有关键操作的日志，涵盖代码生成、环境配置、执行触发等 |
| **用户确认待办列表** | 列出需要人工确认的持久操作点，供下游「开发验收」工位处理 |

### 输出格式约定

- Artifacts 以 YAML/JSON 格式存储于指定路径
- Attempt History 采用追加写入模式，每次执行生成新条目
- 审计日志采用结构化日志格式，便于后续检索与分析

---

## 输入引用

本工位依赖以下上游产物的输入：

1. **子阶段架构评审产物**
   - 经过批准的架构方案文档
   - 接口定义与集成规范
   
2. **项目计划蓝图**
   - 时间线和里程碑要求
   - 资源约束条件

3. **总架构评审产物**
   - 系统级约束和技术栈选型
   - 合规性要求

所有输入均需在开始实施前完成校验，确保版本一致且处于可执行状态。

---

## 质量风险

| 风险等级 | 风险描述 | 防护措施 |
|----------|----------|----------|
| **高** | 代码修改超出子阶段架构范围，导致系统不一致 | 实施前再次核对架构边界，实施中设置代码审查卡点 |
| **高** | Artifact 格式不符合 HarnessOS 规范，导致后续无法执行 | 在提交前对照规范进行自检，使用预定义的模板生成 artifact |
| **中** | 审计日志不完整，影响可追溯性 | 在每个关键操作点强制记录日志，定义最小日志必填字段 |
| **中** | 重跑机制未正确保留历史记录，导致 attempt history 丢失 | 对 attempt history 的写入操作进行原子性保证，存储前后进行校验 |
| **低** | 用户确认触发时机不当，造成流程阻塞 | 与下游工位明确确认节点，在需要确认时主动暂停并通知 |

## 下一工位建议

本工位完成后，建议按以下顺序流转：

### 直接下游：开发验收

**交付内容：**

- 工程任务 Artifact
- Attempt History 记录
- 审计日志

**重点关注：**

- Artifacts 是否可正常执行
- 审计日志是否覆盖所有关键操作
- 重跑机制是否正常工作

### 条件路由

若开发验收发现问题，需要返回本工位进行修复时，应携带具体的返工说明，明确指出问题 artifact 和期望的修正方向。

若涉及架构层面的重大变更，需上报至「子阶段架构评审」工位重新评估，避免在实施层面进行超出范围的修改。
