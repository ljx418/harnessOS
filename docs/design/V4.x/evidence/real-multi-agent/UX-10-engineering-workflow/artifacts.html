# 工位产物：产品规划

## 1. 目标理解

作为产品规划工位，我们的核心职责是将“为 HarnessOS 设计并验证一个可审计的工程任务执行流程”这一宏观目标，转化为具备可操作性的产品需求与业务逻辑。

**核心要点解读：**
*   **Auditability (可审计性)**：这是平台的核心竞争力。要求系统不仅执行任务，还要完整记录“谁、在什么时候、因为什么、执行了什么、结果如何”。这是所有功能设计的基石。
*   **Headless-first (无头优先)**：意味着主要交互发生在 API/CLI 层，但因包含“人工确认”步骤，必须存在清晰、高效的审批管理前台(Frontend/Console)。
*   **Constraint-Driven (受限于约束)**：“不允许自动修改代码”和“用户确认制”是铁律。这定义了平台的本质是 **"Governance Platform" (治理平台)** 而非 purely "Automation Platform" (自动化平台)。

# 工位产物：子阶段架构评审

**角色**：架构  
**工位**：子阶段架构评审  
**场景**：长时工程任务工作流  
**目标**：审查子阶段架构影响  

---

## 1. 目标理解

在子阶段架构评审工位中，架构角色需要验证该子阶段的实现方案是否符合总体架构设计，并评估其对上下游模块及整体系统的影响。具体包括：

- **架构一致性**：子阶段设计是否与总架构评审产出的架构决策保持一致
- **接口契约**：子阶段与其他子阶段/模块的接口定义是否清晰、稳定
- **依赖管理**：外部依赖和内部依赖是否合理，是否引入不必要的耦合
- **非功能性需求**：性能、可扩展性、可维护性、安全性是否符合预期
- **约束合规性**：确认实现方案不违反"不允许自动修改代码"、"不允许自动发布"、"持久操作需用户确认"等核心约束
- **可审计性**：是否为该阶段的产出物建立可追溯的 artifact 和 attempt history

---

本工位完成后，应产出以下 artifact：

| 产物 | 说明 |

| **架构影响分析** | 该子变更对其他模块、系统边界、数据流的影响范围说明 |
| **接口契约定义** | 与上下游交互的 API/协议定义（如有变更） |
| **可追溯性清单** | 关联至总架构评审的决策点，确保每项设计有据可查 |
| **问题跟踪表** | 评审中发现的问题及其状态（未解决/已解决） |

## 4. 质量风险

在子阶段架构评审中需重点关注以下风险：

| 风险类别 | 可能问题 | 影响 |
|----------|----------|------|
| **架构漂移** | 子阶段实现偏离总架构决策，未及时同步更新 | 系统一致性破坏，后期返工成本高 |
| **接口断裂** | 定义的契约与实现不一致，导致下游集成失败 | 开发验收或 E2E 验收阶段阻塞 |

# 开发实施 - 工位产物

## 目标理解

作为开发实施工位，本阶段的核心目标是**将经过评审的子阶段架构方案转化为可运行的工程任务 artifact，并确保整个执行流程具备可审计性和可重跑性**。

具体而言，需要完成以下任务：

- 按照子阶段架构评审确定的方案进行代码实现
- 生成符合 HarnessOS 规范的工程任务 artifact（配置文件、执行脚本、元数据等）
- 确保每个执行步骤都有明确的审计日志
- 实现失败重跑机制，保留每次 attempt 的历史记录

本工位承接来自「子阶段架构评审」工位的输出，并将结果交付给「开发验收」工位进行验证。

---

## 关键输出

| 输出项 | 说明 |
|--------|------|
| **工程任务 Artifact** | 包含完整的配置文件、执行脚本、依赖声明等，用于描述一个可执行的工程任务单元 |
| **Attempt History 记录** | 记录每次执行尝试的元数据，包括时间戳、输入参数、执行结果、失败原因（如有） |
| **审计日志** | 按时间顺序记录所有关键操作的日志，涵盖代码生成、环境配置、执行触发等 |
| **用户确认待办列表** | 列出需要人工确认的持久操作点，供下游「开发验收」工位处理 |

### 输出格式约定

- Artifacts 以 YAML/JSON 格式存储于指定路径
- Attempt History 采用追加写入模式，每次执行生成新条目
- 审计日志采用结构化日志格式，便于后续检索与分析

## 目标理解

作为 HarnessOS Headless-first 工作流平台工程任务中的 **代码检视** 角色，本工位的核心目标是对开发实施阶段产出的代码进行系统性审查，确保代码满足设计规格、质量标准和合规要求，并为后续 E2E 验收阶段提供可执行的准入判定。

本工位在期望流程中处于 **开发验收 → 代码检视 → E2E 验收** 的桥接位置，需验证：

- 代码实现与架构设计的一致性
- 工程任务约束的遵守情况
- 可重跑性与 attempt history 机制的完整性

## 质量风险

基于 HarnessOS Headless-first 工作流平台的上下文，以下是需要重点关注的质量风险域：

### 高风险 (P1)

| 风险项 | 描述 | 可能后果 |
|--------|------|----------|
| **约束“禁止 Agent 自动修改代码”被绕过** | 若代码中存在自动化脚本或机器人直接提交变更 | 违反核心工程伦理，任务合法性存疑 |
| **持久操作缺少用户确认机制** | 若代码实现了未经审批的持久化写入 | 违反“所有持久操作须用户确认”约束 |
| **失败重跑机制缺失** | 若某个子阶段失败后无法恢复执行上下文 | 违反“可重跑并保留 attempt history”要求 |

### 中风险 (P2)

| 风险项 | 描述 | 可能后果 |
|--------|------|----------|
| **Artifact 产出不完整** | 若某阶段未按要求生成可归档的产出物 | 影响审计追溯能力 |
| **人工确认节点遗漏** | 若流程中某处应有人工确认却实现了自动流转 | 违反工作流设计原则 |
| **审计链路断裂** | 若日志/执行轨迹不足以支撑事后审计 | 失去可审计性 |

# 工位产物：端到端验收

## 目标理解

E2E 验收工位的核心目标是验证 HarnessOS Headless-first 工作流平台的完整实现是否满足「可审计的工程任务执行流程」这一定位要求。

需要从端到端的视角审视整个流程链路的贯通性，特别关注：

- 各阶段 artifact 的产出质量
- 约束条件的遵守情况
- 失败恢复与重跑能力
- 用户确认节点的完整性

---

## 关键输出

| 输出项 | 说明 |
|--------|------|
| E2E 验收报告 | 涵盖全流程的功能完整性、合规性、可审计性评估结论 |
| 约束条件检查清单 | 逐项核验 Agent 修改、自动发布、用户确认等约束的执行情况 |
| 流程贯通性鉴定 | 产品规划到人工确认的全链路可追溯性证明 |
| 风险暴露报告 | 识别当前阶段未能解决的已知问题 |

---

## 输入引用

- **原始输入**：工程任务目标及约束定义
- **期望流程**：产品规划 → 项目计划蓝图 → 总架构评审 → 子阶段计划审计 → 子阶段架构评审 → 开发实施 → 开发验收 → 代码检视 → E2E 验收 → 人工确认
- **依赖前置工位输出**：
  - 开发验收的artifact（功能实现物）
  - 代码检视结果（代码质量评估）

---

## 质量风险

1. **artifact 缺失风险**：部分子阶段可能未产出符合规范的 artifact，导致审计链条断裂
2. **重跑能力风险**：若失败阶段的历史记录（attempt history）未被正确保留，会影响可重跑性验证
3. **流程断点风险**：人工确认环节之前的自动化步骤占比需严格控制，防止背离「用户确认驱动」的约束
4. **端到端可追溯性不足**：日志、变更记录、审批痕迹的完整性直接影响审计能力

---

## 下一工位建议

进入「人工确认」工位前，需确保以下前提条件已达成：

- E2E 验收结论为「通过」或「有条件通过」
- 所有约束检查项均已闭环
- 人工确认所需材料（验收报告、artifact清单、重跑记录）已齐备

建议下一工位重点关注：

1. 用户的最终决策点设置是否清晰
2. 各阶段产出的 artifact 是否已完成签收/归档
3. 失败重跑的 attempt history 是否可供回溯审查

# 工位产物：人工确认

## 1. 目标理解

本工位位于长时工程任务工作流的最终核验环节，负责对已完成 E2E 验收的工程产出进行人工审核与确认放行，确保所有持久化操作符合 HarnessOS 平台的审计要求，并为该工程任务画上闭环句号。

## 2. 关键输出

### 人工确认清单

| 序号 | 确认项 | 确认内容 | 结论 |
|:---:|--------|----------|:----:|
| 1 | 代码检视确认 | 代码改动已通过 Code Review，PR 合入规范符合要求 | ☐ |
| 2 | E2E 测试确认 | 端到端测试用例全部通过，功能符合产品规格 | ☐ |
| 3 | Artifact 产出确认 | 各阶段产物齐全且已归档至对应存储位置 | ☐ |
| 4 | 审计日志确认 | 执行轨迹、attempt 历史、变更记录完整可追溯 | ☐ |
| 5 | 失败重跑记录确认（如适用） | 重跑阶段的日志、差异对比报告已留存 | ☐ |
| 6 | 发布准入确认 | 符合「不允许自动发布」约束，仅作待发布就绪标记 | ☐ |

> **确认结论**：□ 通过 / □ 有条件通过（需补充说明）/ □ 驳回

---

## 3. 输入引用

- **前工位产物**：E2E 验收结果报告（含测试用例执行日志）
- **artifact 仓储**：各子阶段产出的设计文档、架构评审纪要、代码变更集
- **审计日志**：从产品规划至 E2E 验收的全链路执行轨迹

---

## 4. 质量风险

| 风险项 | 描述 | 严重程度 | 缓解措施 |
|--------|------|:--------:|----------|
| 人工确认遗漏关键点 | 人工审核可能因疏漏未发现潜在合规问题 | 中 | 对照本清单逐项勾选，留存签字记录 |
| artifact 版本不匹配 | 最终交付物与评审纪要中的版本不一致 | 高 | 核对 artifact 哈希值与 BOM 清单 |
| 审计链断裂 | 部分阶段日志缺失导致不可审计 | 高 | 向前工位发起审计完整性追溯 |

---

## 5. 下一工位建议

本工位完成后，根据确认结论触发以下流向：

- **通过** → 进入发布准备阶段（仅标记就绪，由具备发布权限的角色手动执行发布操作）
- **有条件通过** → 返回开发实施或对应缺陷修复阶段，重新走子阶段流程
- **驳回** → 打回至问题所属阶段，并创建新的 attempt 记录
