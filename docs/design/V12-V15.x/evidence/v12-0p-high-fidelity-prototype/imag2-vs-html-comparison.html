可见数据空间、项目、沙箱、模型、提供商、API 状态、运行状态、证据状态、等待确认。

禁用动作未确认前不得 publish / run；禁用按钮必须显示原因。
验收点1440px 宽度下 chip 不挤压；状态有颜色和文字双重提示。

左侧 Rail + 二级上下文导航

目标：用户理解“当前在哪个产品域”以及“这个域里有哪些对象”。

可见数据一级产品域、当前项目、运行中、证据待审、全部工作流、草稿、已发布、失败/阻断。

验收点至少展示多输入 / 多输出设计；selected、ready、running、blocked、waiting 状态清楚。

目标：用户在一个面板里完成节点审查、配置理解、质量判断和证据追踪。

验收点长文本换行；证据只读；阻断原因必须可见。

底部工作台

可见数据Chat 输入、提案时间线、Trace 事件、质量项、证据 ref、当前 tab。
允许动作输入自然语言、查看 proposal、切换 tab、定位到画布节点。
禁用动作Chat 不能伪装成已执行；未确认 WorkflowDiff 前不得运行。
验收点每个 tab 都有独立目的；空态、加载态、失败态不静默。

目标体验：用户完成 V12 后应当能看懂什么

目标不是“更漂亮的静态图”，而是让用户能快速创建、观察、审查和修改工作流，并理解每个工位的产出与质量。

1
从自然语言进入工作流

用户在底部 Chat 输入目标，系统展示提案时间线和画布草案，而不是静默等待。

2
从画布理解编排结构

用户看到触发器、规划 Agent、并行 Agent、总结 Agent、质检 Agent 的关系与状态。

3
从 Inspector 审查单个工位

用户点击节点后看到角色、目标、工具、MCP、质量门槛、证据和最新输出。

4
从 Trace / Evidence 判断可信度

用户能区分设计参考、fixture、浏览器截图、运行证据和 redacted evidence ref。

5
从确认门槛避免误操作

所有会改变工作流或运行结果的动作都先进入 WorkflowDiff proposal，等待用户确认。

目标体验视觉参考imag2 design reference

典型使用场景：组件如何协同完成任务

以下场景用于后续 V12 浏览器实现和 UX 自动化验收。当前页面只证明设计规格，不证明真实运行。

多 Agent 讨论

  1. 用户输入讨论主题。
  2. 画布生成多个观点 Agent。
  3. 总结 Agent 汇总共识和分歧。
  4. 质检 Agent 检查覆盖度。
  5. Evidence tab 显示讨论证据。

视频分镜创作

  1. 用户输入创意点子。
  2. 规划 Agent 生成镜头结构。
  3. 风格节点固定人物和画风。
  4. 图像节点输出分镜 ref。
  5. 质检节点标记一致性问题。

自然语言修订

  1. 用户要求增加质检或调整顺序。
  2. Chat 生成 WorkflowDiff proposal。
  3. 画布高亮变更节点和边。
  4. Inspector 展示影响范围。
  5. 用户确认前不执行。

失败阻断审查

  1. 节点进入 blocked。
  2. 边和节点显示阻断色。
  3. Inspector 展示原因。
  4. Trace 定位失败事件。
  5. Chat 给出可选修复动作。

证据审计

  1. 用户打开 Evidence tab。
  2. 按节点查看 artifact ref。
  3. 查看质量报告和 runtime report。
  4. 确认 redaction 状态。
  5. 导出审查摘要。

自审结论：当前可进入人工审查，但仍不是实现证据

本轮自审聚焦组件完整性、场景清晰度、设计一致性、证据边界和可实现性。所有视觉图均为 design reference。

UI 一致性：PASS

颜色、圆角、状态 chip、节点卡、Inspector 和底部工作台已统一到 Light Studio 方向。

UX 可解释性：PASS

典型场景说明了用户如何创建、观察、审查、修改工作流，以及如何判断证据可信度。

可实现性：PASS

后续可用 shadcn / lucide / XyFlow / BFF DTO 落地；组件边界没有要求自研底层画布引擎。

False Green：受控

页面明确 design-only，不证明浏览器实现、BFF/DTO、runtime execution 或完整工作流工作台完成。
