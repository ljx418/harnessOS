组件深度设计稿 v4

为什么新增

上一版仍偏整体展示，组件细节不足。v4 单独展开导航、画布、节点卡、Inspector、底部工作台、状态枚举和用户路径。

重点修复

工作流平台作为一级路由；候选节点和模板收纳到二级上下文；节点补多输入多输出端口；连线改为完整曲线；Inspector 与底部工作台分层更清楚。

审查入口

打开组件深度设计稿 v4

该 v4 页面仍是 design_only 原型证据，不能替代 HarnessOS 真实浏览器实现、BFF 日志、DTO snapshot、Playwright trace 或 runtime evidence。

用户可能看到的状态枚举

api_online

顶部显示绿色 API 在线，用户可继续浏览工作台和读取 BFF DTO。

api_offline

顶部转为红色离线，L2 显示缓存只读，Chat 输入显示不可提交原因。

node_hover

节点抬升、端口发光、相邻连线高亮，并展示角色和证据数量提示。

node_selected

节点出现蓝色描边，右侧 Inspector 打开并绑定同一节点 DTO。

invalid_edge

连线变红并回弹，出现策略/能力拒绝原因，不发生任何持久变更。

awaiting_confirmation

底部提案时间线停在确认节点，发布/运行按钮保持禁用或 proposal-only。

evidence_missing

证据入口为 amber，Inspector 展示缺失字段和下一步补证据动作。

blocked

节点或边显示警告徽标，底部 Trace 展示阻断事件和可审计 ref。

关键用户路径

1. 进入项目

用户从 L1 工作台进入项目，在 L2 看到工作流、Agent 和证据对象。

2. 输入目标

用户在底部 Chat 输入自然语言目标，系统生成提案时间线。

3. 观察画布

画布展示 trigger、station Agent、tool/MCP、quality/evidence 节点关系。

4. 审查节点

点击节点打开 Inspector，检查 role、goal、tools、MCP、policy 和证据。

5. 确认或修订

用户在提案时间线中确认、修订或拒绝 WorkflowDiff，不静默发布/运行。
