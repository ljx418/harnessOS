二、组件库：具体样式、状态和开发规格

每个组件都给出可见数据、交互动作、状态枚举和实现备注。浏览器实现阶段应以这些规格映射到 shadcn/Radix/Tailwind/lucide/XyFlow。

目标：能直接指导开发，不再只靠表格描述

全局标题栏 Top Bar

Light Studio 项目 V12 API 在线 等待确认
用途让用户 10 秒内确认 workspace、项目、环境、模型和安全下一步。
状态API 在线/离线、运行中/空闲、证据就绪/缺失、等待确认/已确认。
规则状态 chip 必须可换行，不得挤压标题；主动作最多 2 个。

一级工作台、智能体、工作流、技能/MCP、证据、运行、设置。

交互hover 显示名称，active 高亮，collapsed 仍保留状态数。

布局点阵背景、工具栏、节点、曲线边、端口、小地图、缩放状态。

三、状态枚举：用户会看到什么

状态必须用颜色、文字和原因共同表达，不能只靠颜色。所有失败、等待、禁用、证据缺失都要有可解释文案。

就绪

组件可用，数据已同步，允许查看或继续下一步安全动作。

运行中

Agent 或提案正在生成，必须显示 turn、trace 或 timeline 进度。

等待确认

存在 WorkflowDiff 或高风险动作，必须 confirm/revise/reject。

阻断

策略、权限、证据、配置或质量失败，必须显示原因和修复建议。

只读

V12 画布基础只允许查看与审查，不允许直接 publish/run。

四、典型使用路径与目标体验

下面四条路径用于验收组件是否真的服务用户，而不是只有视觉装饰。

1

进入工作台

用户看到 workspace、项目、环境、API、证据状态，知道当前是只读画布基础。

2

输入自然语言目标

Chat Workbench 生成 proposal timeline，明确不会自动发布或运行。

3

审查画布节点

用户点击节点，Inspector 展示角色、目标、工具、MCP、质量和证据 refs。

4

确认或修订

用户根据质量门槛、证据链和阻断原因决定确认、修订或拒绝 WorkflowDiff。
