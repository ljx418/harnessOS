关键组件细化

以下不是泛泛的“组件列表”，而是每个组件进入 V12-0P 前必须冻结的细节。

component handoff

1. Global Top Bar

显示workspace、project、autosave、API、model、evidence、policy、preview/proposal/run。
状态healthy / degraded / offline / collecting / ready / blocked。
交互状态 chip 可展开原因；禁用按钮必须显示 tooltip。
验收用户 10 秒内知道“在哪、能做什么、为什么不能运行”。

2. Left Product Navigation

一级工作台、智能体、工作流、技能/MCP、证据、监控、设置。
二级当前域下对象列表、计数、最近访问、blocked 标记。
交互rail hover 显示 label；二级导航保持当前域上下文。
验收用户能解释“rail 是产品域、二级是域内对象”。

3. Canvas Workbench

显示只读点阵画布、曲线边、端口、小地图、toolbar、禁用 add/run。
状态empty / read_only / node_hover / node_selected / inspecting / blocked。
交互hover 预览；click 选中；edge hover 显示来源和目标。
验收任何未确认 publish/run 行为都必须阻断并解释。

4. Agent / Station Node Card

显示role、goal、status、model ref、tools、skills、MCP、evidence refs。
IO至少支持多 input / 多 output；端口带 type、required、blocked 状态。
状态idle / running / completed / blocked / failed / missing_config。
验收用户能说清每个工位 Agent 的职责和当前产出。

5. Right Inspector

Tabs配置、输入输出、工具权限、证据、质量、日志。
显示redacted refs、policy decision、quality score、blocked reason。
交互点击证据 ref 打开只读 drawer；质量问题可定位节点。
验收Inspector 只能解释 runtime truth，不构造 runtime truth。

6. Chat Workbench

定位workspace-scoped 独立 chatbox，管理用户空间聊天历史。
输出目标理解、WorkflowDiff proposal、timeline、确认/修改/拒绝。
交互发送后必须有即时状态反馈；长任务显示 streaming/progress。
验收Chat 不能直接 publish/run；只能触发 proposal 或确认流程。

状态细化

V12 前端质感不只是视觉，还包括用户能看懂系统状态。下面这些状态必须在 UI 中有稳定样式。

HealthyAPI、BFF、DTO、evidence 可用，允许预览和提案。
Collecting证据或运行数据收集中，显示 skeleton/progress。
Proposal readyWorkflowDiff 已生成，等待用户确认或修改。
Blocked权限、策略、确认或证据缺失导致动作阻断。
Read-onlyV12 只读画布基础，不能直接编辑 DSL。
Missing config节点缺模型、工具、MCP 或 credential ref。
Failed运行或证据失败，保留 trace 和失败原因。
Evidence ready证据链可审查，仅展示 redacted refs。

从自然语言到提案的可解释路径

用户输入目标后，系统不应静默，也不应直接运行。每一步都要在 Chat、画布和 Inspector 上可见。

1用户输入Chat Workbench 立即显示 input received 和 turn id。
2目标理解显示系统理解、假设、需要确认的风险点。
3生成提案画布出现 proposed nodes/edges，全部标记 proposal。
4证据预览Inspector 展示 policy、quality、evidence refs。
5人工确认用户 confirm/revise/reject；确认前不 publish/run。
边界：本页仍是设计细化证据。它不证明真实浏览器实现、BFF route、DTO snapshot、provider invocation、 runtime execution、完整 Studio 或 Xpert 追平。

V12-0P 冻结前验收表

如果这些细节没有通过，不应进入真实前端实现。

review checklist
验收项 必须看到 失败条件
组件尺度 顶栏、rail、sidebar、canvas、Inspector、Chat 都有稳定尺寸和响应式策略。 按钮挤压、文字溢出、组件高度随状态跳动。
状态样式 healthy、collecting、proposal、blocked、failed、read-only 均有明确视觉语义。 用户看不懂系统是否运行、阻断或等待确认。
画布交互 hover、selected、edge、port、toolbar、disabled action 都可解释。 节点之间没有真实连接感，端口和输入输出不可见。
Agent 可解释性 每个节点能展示 role、goal、tools、skills、MCP、evidence refs 和质量状态。 用户无法判断哪个 Agent 在做什么，产出质量如何。
Chat-to-Proposal 自然语言输入后有即时反馈、提案进度、确认/修改/拒绝路径。 输入后无反馈，或 Chat 被误认为已自动运行工作流。
边界与安全 所有真实运行、发布、写入都要求确认和证据，不把设计稿当 runtime evidence。 任何页面声明产品级前端完成、完整 Studio 或生产可用。
