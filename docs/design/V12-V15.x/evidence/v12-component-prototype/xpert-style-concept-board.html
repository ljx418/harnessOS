V12-0A · Xpert-inspired concept board · design_only

三套更接近产品级工作台的视觉方向。

这份概念板用于解决“当前原型质感不足”的问题。它基于 Xpert 的工作台特征重新推导 HarnessOS 的视觉方向： 顶栏、产品导航、画布、节点、右侧检查器和 Chat Workbench 必须形成一个完整产品界面，而不是孤立组件拼贴。

当前材料仍是设计原型证据，不证明真实浏览器实现、BFF 调用、运行时执行、完整 Workflow Studio 或 Xpert 追平。

本页用于做什么决定

选择方向 从三套视觉密度与产品性不同的方案中，选择 V12-0P 高保真原型主方向。
冻结组件 确认哪些布局、状态、节点、Inspector 和 Chat 关系必须进入后续 Figma 或高保真。
保留边界 本页不能替代浏览器截图、BFF 日志、DTO 快照或 runtime evidence。

三个方向

三套方案都保留 Xpert-like 的核心工作台骨架：左侧 rail、实体二级导航、中心画布、节点卡、右侧 Inspector、 底部或侧边 Chat Workbench。差异在视觉密度、使用人群和最先要打磨的交互细节。

方案对比

这张表用于决定 V12-0P 高保真原型的主方向。建议不要混合所有风格，先冻结一个主线，再把其他方向作为模式或插件化场景。

维度 方案 A：Light Studio 方案 B：Operator Console 方案 C：Creative Studio
首屏观感 最接近 Xpert，轻、清晰、产品化。 更像运维/审计控制台，专业但偏重。 更有创作感，适合视频/分镜演示。
用户上手 低门槛，适合默认入口。 适合高级用户和内部审计。 适合创作者和演示场景。
画布重点 结构、连接、选择、Inspector 联动。 运行状态、trace、policy、风险。 产物预览、素材流、QA Agent。
Chat Workbench 自然语言到 proposal，确认前不运行。 强调运行状态解释和 blocked reason。 强调创作目标、素材修改和质量反馈。
推荐用途 V12-0P 默认方向。 作为“审计/运行模式”保留。 作为“创作模板模式”保留。

建议冻结策略

当前最稳妥的路径不是继续堆视觉效果，而是先冻结主线信息架构，再把不同风格作为模式化扩展。

recommended decision

推荐：以方案 A 作为 V12-0P 主线

方案 A 的默认工作台最接近 Xpert 的产品密度，同时不会牺牲 HarnessOS 的治理边界。 它应冻结为 V12 浏览器工作台的默认信息架构：左 rail、二级对象导航、只读画布、节点卡、右 Inspector、Chat Workbench。

保留：B/C 作为模式和场景

  • 方案 B 进入“运行审计模式”：用于 trace、policy、权限、证据链和失败恢复。
  • 方案 C 进入“创作工作流模板”：用于视频分镜、素材预览、QA Agent 和自然语言优化。
  • 三者共享同一底层组件 token、节点结构、Inspector 数据模型和 Chat proposal 语义。
边界：本页只支持 V12-0A 设计审查和 V12-0P 高保真方向选择。 它不支持跳过 V12-0P、readiness audit、真实浏览器截图、BFF 日志或 DTO 快照。
