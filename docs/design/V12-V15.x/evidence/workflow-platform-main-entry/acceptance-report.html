人类审计步骤

  1. 先阅读 TASKS.md 的 Current Status 与 No-Go，确认本阶段只审 WP-M1A 到 WP-M5A。
  2. 阅读 PRD、目标架构、开发计划、验收门槛和任务矩阵，确认 PV13 是首页体验基线，PV20/PV21 是能力迁移来源。

  4. 读取 browser-action-log.json，核对自动化是否真实执行滚轮缩放、节点拖拽、自由连线和取消连线。
  5. 读取 browser-network-log.json，核对所有浏览器请求是否只落在允许的 BFF route family。
  6. 读取 user-scenario-report.json，核对三个业务场景是否包含真实输入路径、哈希、平台动作、人工审查点和输出证据。
  7. 读取 runtime-inspect-report.json、evidence-panel-report.json 和 agent-executor-integration-report.json，确认 PV21/PV20 能力没有只停留在 UI 文案。
  8. 读取 workflow-platform-capability-parity-report.json，确认从旧入口迁移过来的能力没有被视觉替换导致退化。
  9. 读取 scenario-projection-report.json、business-output-report.json 和 mock-reduction-report.json，确认三业务场景已从路径验收升级为 DTO/evidence 驱动业务产物摘要。
  10. 复跑命令矩阵中的测试命令；若任一命令失败，打回开发或验收计划阶段。
  11. 检查非声明边界；不得把本报告解释成外部 App 接入、生产部署或完整商业交付完成。

目标架构与当前实现

目标架构要求工作流平台成为首入口，并将画布、WorkflowDiff、发布、运行、Human Gate、Evidence Review 与受治理 Agent/Tool/Skill/MCP 资源收敛到同一产品路径。当前实现复用 PV21 BFF DTO 作为工作流闭环，复用 PV20 BFF DTO 作为受治理执行证据，不新增绕过 BFF 的浏览器调用。

架构层目标当前实现
Browser entry默认进入 PV13 Light Studio 工作流平台。App.tsx 与 WorkflowStudioLayout.tsx 将根入口和 workflow-platform 映射到 V13EditableStudio。
Canvas / Workbench力感画布、节点、端口、连线、Inspector 和底部审查区可操作。V13EditableStudio.tsx 与 v13-editable-studio.css 提供 PV13 基线体验，并记录浏览器动作日志。
BFF / DTO浏览器只通过 BFF DTO route 与后端交互。/bff/v13/*、/bff/pv21/*、/bff/pv20/*、/bff/workflow-platform/* 通过网络日志验证。
Runtime / Evidence运行、人工门禁、证据审查在同一工作台可理解。通过 PV21/PV20 compatibility DTO 生成 runtime inspect、evidence panel 和 executor integration 报告。

命令矩阵记录的是本轮自动化验收命令；若环境变化，需要复跑。低命令已列出，可复现；失败应打回开发或验收计划阶段。
